== Why CI carries the lint gates

The development host runs a distro Rust (`/usr/bin/{rustc,cargo}` 1.85) with
**no rustup, and therefore no `rustfmt` or `clippy`**. Those gates cannot run
locally, so CI is the first and only place they are enforced. `cargo test` and
`cargo build` *are* run locally and are expected green before every commit.

== Workflow: `.github/workflows/ci.yml`

Two jobs, least-privilege (`permissions: contents: read`):

* *lint* (stable): `cargo fmt --all -- --check` and
  `cargo clippy --workspace --all-targets --locked -- -D warnings`. clippy runs
  rustc lints too, so it also enforces the warning-free build for workspace
  crates. The full suite's `no-<script>` SSR assertions and the git
  no-data-loss harness run under *test*, so `cargo test` is the security/safety
  gate.
* *test* (matrix `1.85.0` + `stable`): `cargo build --locked` and
  `cargo test --workspace --locked`. `1.85.0` pins the declared MSRV
  (`rust-version` in the workspace); `stable` catches what the pinned toolchain
  misses. `--locked` fails on `Cargo.lock` drift (the lock freezes the
  rustc-1.85-compatible comrak resolution).

Actions: `actions/checkout@v5`, `dtolnay/rust-toolchain@{stable,master}`,
`Swatinem/rust-cache@v2`. Dependabot (`.github/dependabot.yml`) keeps these and
the Cargo deps current.

== Before the first push to a remote

. rustfmt cannot run on this host, so the code is *not yet* rustfmt-normalised.
  On a rustfmt-capable machine run `cargo fmt --all` and commit, or expect the
  first *lint* run to fail on formatting only.
. Pushing `.github/workflows/**` needs a token with the `workflow` scope; the
  default `gh` token does not have it. Run `gh auth refresh -s workflow` first.
