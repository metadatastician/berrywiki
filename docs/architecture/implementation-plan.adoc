== Executive summary

BerryWiki's hardest layers are already green and committed: `berrywiki-core`
(zero-dependency, I/O-free engine, 35 tests), `berrywiki-store` (`WikiStore`
trait + `LocalFolderStore` with create/update/move/delete/attachments/sidebar,
commit `9f411fe`), and `berrywiki-git-compat` (git-CLI evidence harness with
six no-data-loss conflict tests, commit `a47e667`). Several of the candidate
designs believed the harness was uncommitted; it is not, and this plan does not
re-schedule that work.

The plan drives at a single product gate: *the owner browses, edits, and safely
syncs their real `.wiki.git` from BerryWiki daily*. Five phases get there:

. *Read your real wiki* — read-only three-pane SSR explorer + repo
  publication with estate CI + the credential-gated *read-side* live spike.
. *Edit in BerryWiki* — full authoring loop (editor, preview, drafts,
  create/delete, transactional move/rename with inbound-link rewriting) while
  the user keeps syncing with terminal git. A genuinely usable increment.
. *Sync from the app* — commit/pull/push with the safety rules enforced by
  construction, the conflict workflow, and the *write-side* live spike.
  *This is the daily-use gate.*
. *Organise it* — tags, history, attachments, templates, accessibility pass.
. *CherryTree import, packaging, MVP declaration.*

SPARK proofs, the AffineScript→typed-wasm enhancement layer, Zim import and a
gitoxide re-evaluation are explicitly post-MVP, with the ground prepared (not
paid for) during the MVP: a numbered invariants ledger from Phase 1, pure
plan/apply seams, and golden vector corpora.

== Decisions taken

Locked decisions from ADR-0001..0004 are taken as given and not re-litigated.
New expensive-to-reverse decisions are marked *ADR required* and get their own
ADR before or in the slice that implements them.

[cols="2,3,4,2", options="header"]
|===
| Decision | Choice | Rationale (and contradictions resolved) | Reversibility

| UI now
| Zero-JS server-side-rendered web app: axum 0.8 + maud templates + comrak
  (GFM) rendering. Single binary `berrywiki serve <path\|url>` on localhost;
  reachable from the Windows browser via WSL mirrored networking. Three panes:
  tree \| rendered page \| outline + backlinks. *ADR-0005 (this decision) —
  see link:../decisions/0005-ui-technology.adoc[].*
| The daily question is "does this render the way GitHub will" — that needs
  HTML, ruling out a TUI as primary (two designs proposed TUI-first; both were
  marked down by judges for exactly this). Rust→wasm frameworks are rejected
  now: wasm-bindgen ships generated JS glue (doctrinally unresolved) and the
  host toolchain (distro Rust 1.85, no rustup) cannot target wasm32 locally —
  the one design that bet on Leptos was judged fatally flawed on both counts.
  Zero-JS SSR has no doctrine ambiguity, the best accessibility story
  (semantic HTML + forms), the smallest dependency tree, and a thin swappable
  seam for AffineScript. CI enforces the choice mechanically: a test asserts
  *no `<script>` element in any HTTP response*.
| High — templates are thin; all behaviour lives behind HTTP endpoints, so a
  future front-end swap does not touch the engine.

| Git engine
| Shell out to the git binary via a new `berrywiki-git` crate: a *closed
  command enum* rendered to argv (no string splicing), parsing only
  `--porcelain=v2` / `-z` output, `LC_ALL=C`, `GIT_TERMINAL_PROMPT=0`,
  minimum git version check (floor 2.30), repo-level advisory lockfile.
  *ADR required (ADR-0006), written in Phase 3 with harness evidence
  attached.*
| Unanimous across all five designs; the wrapper hardening is grafted from the
  verification-first design (judges' strongest git encoding): an audit test
  enumerates every constructible command and asserts `--force`,
  `--force-with-lease`, `reset --hard` cannot appear — force-push is
  *unrepresentable*, not merely forbidden. A CI grep gate over `src/`
  (risk-first graft) backs this up cheaply. The CLI matches byte-for-byte
  what a user without BerryWiki does (the product's core promise), and all
  evidence from the committed `berrywiki-git-compat` harness transfers
  directly. libgit2 rejected (C dependency, divergent merge behaviour,
  CVE history); gitoxide's push/merge surface still maturing — re-evaluate
  post-MVP behind the same typed interface.
| Medium — the typed interface isolates the engine; gitoxide could replace
  the subprocess behind it later.

| Credential handling
| Fine-grained PAT (Contents R/W, single wiki repo), sourced from
  reasonably-good-token-vault, exported as `BERRYWIKI_GITHUB_TOKEN`. Injected
  via *GIT_ASKPASS self-re-exec*: the `berrywiki` binary re-invoked with a
  hidden flag that prints the env var — token never on argv, never in
  `.git/config`, never in a URL. All captured git output passes a redaction
  layer; a leak-hunting test greps every log artifact for the token.
| Estate-Native graft, endorsed by all three judges: exceeds ADR-0002's floor
  with no helper-script file to manage.
| High — swap of injection mechanism touches one module.

| Search
| `SearchIndex` trait in a new `berrywiki-index` crate. MVP implementation:
  in-memory index (title > headings > tags > body ranking, case-folded,
  code-fence-aware via core's extractors), rebuilt on store reload —
  milliseconds at real wiki scale. SQLite FTS5 (rusqlite, `bundled`) lands
  behind the *same trait* only if the Phase 5 perf run on a 2,000-page
  synthetic wiki shows measured pain; if it lands, the index lives in the
  *XDG cache* keyed by repo hash — never inside the wiki clone — with a
  versioned schema and a `berrywiki reindex` full-rebuild command, and the
  in-memory implementation is *kept permanently as a differential-test
  oracle*.
| Resolves the designs' three-way split (in-memory vs FTS5-now vs FTS5 with
  in-memory oracle): defer the schema/versioning machinery past the daily-use
  threshold without foreclosing it; take the oracle pattern
  (verification-first graft) and the never-pollute-the-clone location
  (Estate-Native graft, protects "usable without BerryWiki"). Tantivy
  rejected unanimously: dependency weight disproportionate to wiki scale.
| High — trait seam proven with two implementations before FTS5 exists.

| Transactional move/rename
| Two-layer design: a *pure ChangeSet planner* in `berrywiki-core`
  (`plan_move`/`plan_rename` → ordered `FileOp` list as plain data: validate
  parent → cycle check → metadata/position edits → filename rename →
  surgical inbound-link span rewrites → attachment references → sidebar
  regeneration), plus a *journaled apply* in the store: write
  `.berrywiki/journal.json` listing every FileOp *before touching anything*;
  apply via atomic temp-write+rename; on failure or crash, roll back by
  `git restore` of exactly the journaled paths (git is the transaction log).
  Recovery rules on startup: journal present + no commit → restore journaled
  paths from HEAD; journal present + commit exists → delete journal. One
  logical operation = one commit, sidebar included.
| Merges the three designs' mechanisms: pure planner (risk-first/product-ux),
  journal protocol with explicit crash-recovery rules (risk-first S4.2 —
  judges called it "a complete, testable never-half-moved mechanism"), and
  git-restore-from-HEAD rollback (Estate-Native — "the simplest sound
  transactional-move design offered"). Link rewrites are surgical span edits
  computed by the existing code-fence-aware extractor — never regex over raw
  Markdown. Case-only renames done as a two-step rename for case-insensitive
  filesystems.
| Medium — the planner is data-in/data-out and is the future SPARK target.

| Conflict policy
| Pull is `merge --ff-only` when Behind; when Diverged, an explicit conflict
  flow: conflicts classified into a machine-readable `ConflictReport` with
  base/ours/theirs extracted via `git show :1:/:2:/:3:<path>`.
  `_Sidebar.md` conflicts are *auto-resolved by regenerating from the merged
  graph* (the harness already shows they behave like ordinary file
  conflicts). *Metadata-block and body conflicts are always surfaced to the
  user — never silently merged.* Modify/delete keeps local content plus a
  diagnostic. Abort is always available and provably restores the pre-merge
  working tree (already harness-tested).
| Grafts risk-first's classification and sidebar auto-regeneration (judges:
  "turns the most common conflict class into a non-event") but *rejects* its
  silent per-key union of metadata conflicts, which one judge flagged as
  edging the never-discard-local-work line. Resolution: only the sidebar —
  which is derived data by definition — auto-resolves.
| High — policy layer above the git wrapper.

| Drafts
| Autosaved on idle to the XDG state dir, keyed by repo hash + page id —
  never inside the clone, never committed. Visible save-state in the UI;
  restore-on-open banner; drafts are additive-only (never deleted without
  explicit user action); a kill-mid-edit test proves an unsaved draft
  survives.
| Product-ux graft (judges: "the trust-building feature for a refugee whose
  old tool never lost a keystroke"), adapted to SSR: autosave is a periodic
  form post in the MVP, upgraded to as-you-type in the enhancement era.
| High.

| Live spike timing
| Split: *read-side* evidence (metadata invisibility, link matrix, sidebar
  depth, nested dirs, default branch) in Phase 1 before any editing exists;
  *write-side* evidence (push, non-FF rejection, conflicts end-to-end, auth
  redaction, private clone) in Phase 3 before in-app sync is declared
  working. Credential-gated throughout; see <<spike>>.
| Each assumption is verified just before it becomes load-bearing — endorsed
  by all three judges over both "spike-first-and-nothing-else" (risk-first
  Phase 1, judged to withhold usable product needlessly) and "spike at
  Phase C after two build phases" (verification-first, judged late).
| n/a — evidence, not code.

| Estate CI timing
| One bounded PR in Phase 1, alongside repo publication: the 4 required
  hyperpolymath/standards workflows (*names verified against
  `standards/.meta` at push time* — known doc drift), `hypatia-scan` with
  `--exit-zero`, secret scanner, community-health files as `.md` with the OU
  address, `.a2ml` meta files (not `.scm`), no third-party AI review bots,
  all actions SHA-pinned per action-trust-layers. Push requires
  `gh auth refresh -s workflow` first.
| Resolves the judges' tension: one judge dinged the base design for
  deferring standards CI to Phase 5; another punished a whole
  estate-citizenship phase before any product. Resolution: estate CI is
  cheap and bounded — do it in one PR early, and do *nothing else*
  infrastructure-shaped before the user can open a wiki.
| n/a.

| SPARK insertion point
| Post-MVP, never on the MVP critical path. Prepared from Phase 1 via a
  numbered invariants ledger (`docs/proofs/invariants.adoc`, INV-1..6)
  cross-referenced to Rust tests, with *demote-or-gate language discipline*:
  every invariant reads "tested, proof scheduled" until a green gnatprove CI
  lane gates that specific claim. See <<spark>>.
| All designs except verification-first put SPARK post-MVP; two judges
  independently flagged verification-first's gnatprove-inside-the-MVP-gate
  as the proof-before-product pattern the estate's own theatre-cleanup
  doctrine warns against. The ledger + language discipline (Estate-Native
  graft) keeps the door genuinely open at near-zero cost.
| n/a.

| Generated-JS-glue doctrine question
| *Parked, not ruled.* The SSR MVP ships zero JS of any provenance, so
  nothing depends on the answer. A formal user ruling (*ADR required,
  ADR-0007*) is the gate for starting the post-MVP AffineScript-era
  enhancement layer (drag-and-drop, inline autocomplete, live preview).
| Resolves the contradiction between the risk-first design (build a
  timeboxed wasm spike pre-MVP to force the ruling) and the base design
  (park it): the spike is *unbuildable on the local toolchain* (no rustup →
  no wasm32), so pre-MVP it would be CI-only guesswork. The ruling is a
  decision for the user, not an artifact to build; it is scheduled as an
  explicit open question with a deadline of "before the enhancement layer
  starts".
| n/a.

| Drag-and-drop
| Not in the MVP. Moves are form/keyboard operations with a dry-run diff
  preview (files to be touched, links to be rewritten) before confirm. When
  pointer DnD ships in the AffineScript era, it MUST drive the *identical
  plan/apply path* as the keyboard/form flow, with a differential test
  asserting both input methods produce byte-identical commits.
| The dual-input-one-path contract is a product-ux graft all three judges
  recommended; recording it now costs nothing and prevents a second move
  implementation later.
| n/a.

== Crate layout

Workspace, existing members kept unchanged in mission:

|===
| Crate | Role

| `berrywiki-core` (exists)
| I/O-free deterministic engine; *zero runtime dependencies forever*,

| Test-support git sandbox (bare remote + two clones, hermetic config), six

| `berrywiki-render` (new, Phase 1)
| comrak GFM Markdown→HTML; heading-anchor slugs reusing core's slug
  algorithm; `[[wikilink]]` → resolved `<a href="/page/:id">` via the
  PageGraph. Keeps core zero-dep. Golden-HTML tests over the fixture wiki.

| `berrywiki-index` (new, Phase 1)
| `SearchIndex` trait + in-memory implementation; optional `fts5` feature
  later adds the rusqlite implementation (see <<decisions>>).

| `berrywiki-git` (new, Phase 3)
| Production sync engine: closed command enum → argv, porcelain-v2 `-z`
  parsers, lockfile, GIT_ASKPASS self-re-exec, redacting logger, SyncState
  computation. Integration-tested exclusively against the
  `berrywiki-git-compat` sandbox (no network).

| `berrywiki-import` (new, Phase 5)
| CherryTree `.ctd` (XML, quick-xml) and `.ctb` (SQLite, rusqlite behind a
  feature) → neutral `ImportModel` → store. Dry-run default; machine + human
  reports; source opened read-only, byte-hash-verified untouched. Zim reuses
  the model post-MVP.

| `berrywiki-app` (new, Phase 1)
| The axum + maud SSR server and the `berrywiki` CLI (`serve`, `check`,
  `sidebar`, later `reindex`, `import`, `spike`). Composes everything; the
  only crate that knows about HTTP. `GitWikiStore` (Phase 3) composes
  `LocalFolderStore` over a managed clone directory — no `WikiStore` trait
  changes needed.

== Phase-by-phase plan

=== Phase 1 — Read your real wiki (read-only explorer + read-side evidence)

*Goal:* the user opens a manually-cloned real `.wiki.git` in a browser daily
for navigation and reading, and the provisional ADRs (0001 filenames, 0004
metadata invisibility) get real GitHub evidence before anything is built on
them.

*Slices (each PR-sized):*

. *S1.1 Publish + estate CI (one PR).* Push the four existing commits to
  `github.com/metadatastician/berrywiki` (user runs the push;
  `gh auth refresh -s workflow` first). Add: `rust.yml`
  (dtolnay/rust-toolchain pinned by SHA: `cargo fmt --check`,
  `clippy --all-targets -D warnings`, `cargo test`; separate MSRV 1.85 job) —
  lint is CI-only since the WSL host has no rustfmt/clippy; the 4 required
  hyperpolymath/standards workflows with names verified against
  `standards/.meta` at push time; `hypatia-scan` with `--exit-zero`; secret
  scanner; a grep gate forbidding `--force`, `--force-with-lease`,
  `reset --hard` under `src/`; the core-stays-zero-dep gate;
  community-health `.md` files (CONTRIBUTING, SECURITY, CODE_OF_CONDUCT with
  `j.d.a.jewell@open.ac.uk`); required `.a2ml` meta files. No third-party AI
  review bots, ever.
. *S1.2 Invariants ledger.* Create `docs/proofs/invariants.adoc`: INV-1
  metadata round-trip idempotence; INV-2 hierarchy acyclicity + deterministic
  sibling order; INV-3 sidebar determinism; INV-4 never-lose-local-work;
  INV-5 never-half-moved; INV-6 path safety (root containment). Each entry
  cross-references its Rust tests and reads "tested, proof scheduled".
. *S1.3 `berrywiki-render`.* comrak with GFM options (tables, strikethrough,
  tasklist, autolink); heading-anchor slugs shared with core;
  wikilink resolution through the graph; golden-HTML tests over the fixture.
. *S1.4 `berrywiki-app` skeleton.* `berrywiki serve <folder>` (axum + maud +
  tokio, zero client JS); routes: `/` (tree + Home), `/page/:id` (three-pane),
  `/diagnostics`. Verified against `fixtures/test-wiki` AND a manual clone of
  one of the user's real wikis. Closes Phase-0 leftover item 10.
. *S1.5 `berrywiki-index` + `/search`.* `SearchIndex` trait, in-memory
  implementation, search form + ranked results with per-hit context snippets.
. *S1.6 Live spike part A (read-side, credential-gated, user-in-loop).* Run
  the read-side experiment set from <<spike>> against a disposable wiki.
  Flip ❓ rows in the compatibility report to ✅/❌ with dates; confirm or
  revise ADR-0001 and ADR-0004 on evidence.

*Exit criteria:*

* `cargo test` green in CI including all `berrywiki-git-compat` scenarios;
  fmt/clippy and all estate lanes enforced on every PR.
* User navigates a real cloned wiki in the browser: tree, rendered pages,
  outline, backlinks, diagnostics, search — with an automated test asserting
  *no `<script>` element in any HTTP response*.
* Compatibility report contains dated live evidence for every read-side row;
  ADR-0001 and ADR-0004 moved from Proposed/Provisional to Accepted, or
  revised with the recorded fallback chosen.
* Repo public with all CI lanes green; invariants ledger exists.

*Test requirements:* golden-HTML fixture tests (render); in-process HTTP
tests via `tower::ServiceExt` asserting on returned HTML — no browser
automation needed because there is no JS; proptest suite for INV-1 (metadata
round-trip/idempotence/unknown-preservation/never-panic) with failure seeds
committed.

=== Phase 2 — Edit in BerryWiki (user pushes with plain git)

*Goal:* BerryWiki replaces the GitHub web editor for authoring; the user
keeps using terminal git for sync. A genuinely usable increment that
de-risks the editor before sync exists.

*Slices:*

. *S2.1 Edit loop.* `GET /page/:id/edit` (source textarea; source is
  canonical), POST preview (server round-trips through `berrywiki-render`
  into the same page shell), POST save via `LocalFolderStore::update_page`.
  Round-trip test: saving unchanged source produces a zero-byte diff.
. *S2.2 Drafts.* Idle-autosave form posts to the XDG state dir keyed by
  repo hash + page id; visible save-state; restore-on-open banner;
  kill-mid-edit recovery test; `berrywiki drafts` list/GC subcommand.
. *S2.3 Create + delete.* `/create` form (title, parent picker, optional
  template body); delete for leaf pages with confirm step (store's
  has-children refusal surfaced); sidebar regenerated in the same logical
  operation.
. *S2.4 ChangeSet planner + journaled apply (the one real engine gap).*
  Pure `plan_move`/`plan_rename` in `berrywiki-core` computing surgical span
  edits from the existing code-fence-aware link extractor; journaled apply
  in the store per <<decisions>> (journal written before any mutation;
  git-restore rollback; startup recovery rules). Property tests: applying
  any generated plan preserves acyclicity, id/filename uniqueness, and
  resolves every previously-resolving link (INV-5 ledger entry updated).
. *S2.5 Move/rename UI.* Form with new-parent select + position + optional
  retitle, showing a *dry-run diff preview* (files to be touched, links to
  be rewritten) before confirm. Case-only renames two-step.
. *S2.6 Link insertion helper.* `<datalist>`-backed page picker on the edit
  screen emitting the correct link form (per Phase 1 spike evidence).
  Honest scope note: `<datalist>` attaches to an `<input>`, so this is a
  side-picker with copy-in, not inline completion inside the textarea —
  inline completion is an enhancement-era item.

*Exit criteria:*

* User authors real wiki content in BerryWiki daily and pushes via terminal
  git; GitHub renders results correctly (spot-checked against spike
  evidence — *the general rendering claim remains unverified beyond the
  spiked rows*).
* Editing an unrelated page never churns other files or metadata bytes
  (byte-diff test in CI).
* Move of a page with 10+ inbound links across the fixture rewrites every
  link, regenerates the sidebar, and leaves zero new diagnostics — as one
  journaled transaction with a tested mid-failure abort path (crash injected
  between each FileOp).
* An unsaved draft survives a killed process and is offered for restore.

*Test requirements:* store-level integration tests on temp dirs;
crash-injection matrix over the journal protocol; property tests for planner
invariants; byte-stability golden tests; "hostile edit" tests (mangle
metadata, hand-rename files) proving graceful degradation to diagnostics.

=== Phase 3 — Sync from the app (THE daily-use gate)

*Goal:* full read-edit-commit-pull-push loop inside BerryWiki against real
public and private GitHub wikis, with the no-data-loss rules enforced by
construction and live-verified. After this phase the user has stopped using
GitHub's editor entirely.

*Slices:*

. *S3.1 `berrywiki-git`.* Closed command enum → argv; porcelain-v2 `-z`
  parsers (property-tested); `LC_ALL=C`, `GIT_TERMINAL_PROMPT=0`, minimum
  git 2.30 startup check; repo-level advisory lockfile serialising all
  mutating ops; GIT_ASKPASS self-re-exec token injection; redaction on every
  error path. *Audit test:* enumerate all constructible commands, assert no
  forbidden flag can appear. Entire surface integration-tested against the
  `berrywiki-git-compat` sandbox. Write *ADR-0006 (git engine)* with harness
  evidence attached. CI gains a two-git-version matrix (2.30-era + latest).
. *S3.2 `GitWikiStore` + SyncState.* Composes `LocalFolderStore` over a
  managed clone (`~/.local/share/berrywiki/clones/<host>/<repo>` on first
  `serve <https-url>`). Explicit SyncState (Clean/Dirty/Ahead/Behind/
  Diverged/Conflicted) computed from fetch + ahead/behind; push reachable
  only from Ahead — fetch-before-push enforced by the type surface, not
  convention. Status strip in the UI.
. *S3.3 Commit workflow.* `/changes` page (porcelain-parsed change list;
  regenerated `_Sidebar.md` grouped with the operation that caused it),
  message box, atomic logical commit. Rule: tree operations and their
  sidebar regeneration always land in the same commit (asserted by a
  harness test inspecting the commit).
. *S3.4 Sync operations + conflict flow.* Fetch button with remote-change
  banner; Pull as `merge --ff-only` when Behind; Diverged routes to
  `/conflicts`: `ConflictReport` classification (base/ours/theirs via
  `git show :1:/:2:/:3:`), sidebar conflicts auto-regenerated from the
  merged graph, body/metadata conflicts shown side-by-side with pick-ours /
  pick-theirs / hand-edit resolution → resolve → commit; abort always
  available, provably restoring the pre-pull tree. Force-push is not
  constructible anywhere.
. *S3.5 Live spike part B (write-side, credential-gated).* Run the
  write-side experiment set from <<spike>>. Move ADR-0002 to Accepted on
  evidence. Wire the credential-gated `#[ignore]` suite into a
  `workflow_dispatch`-only CI job with an environment-scoped secret (never
  on fork PRs).

*Exit criteria:*

* *DAILY-USE THRESHOLD MET:* user connects to at least one real private wiki
  by URL and performs browse → edit → commit → pull → push entirely in
  BerryWiki *for a week* without touching terminal git or the GitHub editor.
* Every sandbox conflict scenario passes through the production
  `berrywiki-git` path (not just the harness), each asserting local work
  survives (INV-4).
* Force-push unrepresentability audit test and token-leak-hunting test green.
* Live write-side evidence recorded with dates; ADR-0002 Accepted;
  non-fast-forward push and a same-page conflict handled end-to-end in the
  UI against real GitHub with no lost edits. Until that run, all of this
  remains *unverified*.

*Test requirements:* sandbox-only integration for the whole engine;
model-style interleaving tests (local edits vs remote pushes vs sync
commands across two sandbox clones; invariant: every locally-authored byte
remains reachable in the working tree, a commit, or an explicit conflict
artifact); in-process HTTP e2e for the full commit→pull→conflict→resolve
flow; credential-gated live suite.

=== Phase 4 — Organise it (finish the MVP checklist minus import)

*Goal:* close the remaining MVP items — tags, history, attachments,
templates — plus the accessibility/keyboard pass.

*Slices:*

. *S4.1 Tags.* Tag editing in the page edit form (metadata already models
  tags), `/tags` index, tag filter on the tree pane and in search ranking.
. *S4.2 History view.* `/page/:id/history` from `git log --follow` (survives
  BerryWiki renames); render any historical version read-only; unified diff
  between versions (server-rendered `<pre>`, no JS).
. *S4.3 Attachments.* Upload form → `add_attachment`
  (`assets/<page-id>/…`, already traversal-proof); insert-link helper
  emitting relative Markdown; app serves `assets/` locally. *Live spike
  addendum:* verify attachment links render on GitHub's wiki UI —
  *unverified until run*; settles the pending attachments ADR row
  (*ADR required* on the evidence).
. *S4.4 Templates.* Built-in defaults + user templates from
  `.berrywiki/templates/*.md` (excluded from the page graph), selectable at
  create time. (Whether an in-clone dot-directory pollutes GitHub's page
  list is a spike row — *unverified*.)
. *S4.5 Accessibility + keyboard pass.* Landmarks, skip-links, focus order,
  accesskeys for pane switching/edit/save/search, all forms labelled;
  automated landmark/label assertions on rendered fragments in CI; a manual
  NVDA smoke script documented in `docs/a11y/` (NVDA only — no macOS device
  exists in this environment, so no VoiceOver gate).

*Exit criteria:*

* Every MVP checklist item except CherryTree import demonstrably works
  against the user's real wiki.
* Attachment row in the compatibility report carries dated evidence.
* Keyboard-only walkthrough of the full daily loop (navigate → edit →
  commit → push) succeeds with no pointer use.

=== Phase 5 — CherryTree import, packaging, hardening — declare MVP

*Goal:* migrate the user's actual CherryTree notebooks in, make the binary
reproducibly installable, complete hardening, and formally declare the MVP.

*Slices:*

. *S5.1 `berrywiki-import` (read side).* `.ctd` XML first (quick-xml), `.ctb`
  SQLite behind a feature (rusqlite) → neutral `ImportModel`;
  rich-text→Markdown downconversion with an explicit lossiness table; source
  opened read-only, byte-hash-verified untouched.
. *S5.2 Dry-run.* `berrywiki import --dry-run <file>` and an `/import` page
  producing a machine report (JSON: node count, depth, losses, collisions)
  and a human Markdown report; nothing written. Fuzz both importer
  front-ends (never panics; always report-or-diagnostics).
. *S5.3 Apply.* Pages created through the store (UUIDv7 ids, parent chains,
  positions, images → `assets/<id>/`), one atomic logical commit,
  idempotency marker preventing accidental double-import (import twice ==
  import once, proven by test). Acceptance: import one of the user's *real*
  CherryTree notebooks.
. *S5.4 Packaging.* `guix.scm` (primary; cargo-build-system; git as
  propagated input) building `berrywiki-app`; `flake.nix` fallback; CI gets
  a `nix flake check` lane as the pragmatic verifier while `guix shell -f
  guix.scm -- cargo test` is exercised locally; README quickstart;
  `cargo install --path crates/berrywiki-app` documented as the plain path.
. *S5.5 Hardening + perf.* cargo-deny/cargo-audit lane; security pass
  re-running path-traversal and redaction suites plus a review of the upload
  path; performance sanity run on a synthetic 2,000-page wiki — only if
  reload/search measurably lags does the FTS5 `SearchIndex` implementation
  (+ versioned schema + `berrywiki reindex` + differential oracle tests)
  land now; otherwise it is recorded as a post-MVP trigger with the
  measurement attached.

*Exit criteria:*

* A real CherryTree notebook imports cleanly: dry-run report reviewed, apply
  produces a browsable, GitHub-renderable tree in one commit, source file
  hash unchanged.
* `guix shell -f guix.scm` builds and runs the binary; nix build verified
  once.
* Full CI green including the manual-dispatch live job.
* *MVP declared:* every checklist item verified against real wikis, the
  compatibility report fully evidence-backed. Post-MVP backlog recorded:
  Zim import, AffineScript enhancement layer (gated on ADR-0007), FTS5 at
  scale, gitoxide re-evaluation, SPARK proofs (see <<spark>>), dual-input
  DnD contract.

[#spark]
== SPARK proof roadmap

GNATprove enters *post-MVP*, never on the MVP critical path — proof effort does
not advance daily use, and the estate default "Rust → Rust/GNATprove" is
explicitly opportunistic. Two judges independently flagged the one design
that put gnatprove inside the MVP gate as the proof-before-product pattern
the estate's own SPARK-theatre cleanup exists to prevent. The MVP prepares
the ground without paying for it:

* *Invariants ledger from Phase 1* (`docs/proofs/invariants.adoc`, INV-1..6)
  cross-referenced to Rust tests, under the *demote-or-gate* discipline: no
  "proven" or "verified" language anywhere in docs or README until a green
  gnatprove CI lane gates that specific claim; until then the ledger says
  "tested, proof scheduled".
* Proof-friendly structure for free: core stays zero-dep and pure; the
  ChangeSet planner is data-in/data-out with no closures in its interface;
  frozen golden vectors (metadata round-trips, sidebar outputs, planner
  outputs) accumulate as the future differential corpus.

*Post-MVP insertion order* (GNAT 14.2 / gnatprove 14.1.1 via Alire, already
provisioned on the WSL host):

. *Metadata block grammar* (INV-1): SPARK reference implementation with
  proved parse/serialise round-trip, idempotence, termination; AoRTE
  (silver) minimum, gold contracts on the named lemmas.
. *Path-safety kernel* (INV-6): `validate_component`/root-containment from
  `berrywiki-store/paths.rs` — small, security-critical, pure.
. *Forest builder* (INV-2): proved acyclicity + total deterministic sibling
  order.
. *Move state machine* (INV-5) and the git argv constructor (supports INV-4)
  as the ambitious later targets.

*Mechanism:* not a port of the crates. Small SPARK reference implementations
in `proofs/spark/`, bound to the Rust engine by a CI differential harness
demanding byte-identical outputs over the shared golden corpus plus
property-generated inputs — proofs of the specification, plus continuous
evidence the Rust implementation refines it. The vcl-ut FFI swap-in pattern
(shipping the proven SPARK parser behind Rust FFI) is recorded as an option,
not scheduled. Full-engine SPARK translation is explicitly *not* planned.

[#critique-appendix]
== Critique appendix (adversarial review, unresolved)

Three critics (feasibility, doctrine, spec-coverage) reviewed the synthesis.
The revision that would fold these in did not run. Findings are consolidated
below (near-duplicates merged); each must be resolved before the affected
slice is executed.

=== Major — must resolve before the slice is built

. *Zero-JS idle-autosave is infeasible.* A zero-`<script>` SSR app has no
  client mechanism to detect idleness or auto-submit a form, so the Phase-2
  exit criterion "an unsaved draft survives a killed process" and the
  never-lose-a-keystroke promise cannot hold as written. Options: explicit
  Save/Save-draft actions with clear unsaved-state indication; accept that
  drafts are server-side per submit; or reopen the client-script question via
  ADR-0007 *before* promising autosave. Do not ship a trust-critical feature
  that cannot work.
. *Non-leaf move/rename is a subtree cascade.* Under ADR-0001 flat
  `Parent--Child.md` filenames, moving or renaming a page changes the filename
  of *every descendant*, not just the page. The move planner design, its test
  matrix and the effort estimate model only the single moved page. Re-scope
  the transactional-move slice to a subtree operation (validate → compute all
  descendant renames → rewrite all inbound links → single commit) with tests
  over multi-level subtrees.
. *Journal rollback can discard local work (INV-4 breach).* Recovering a
  half-applied operation via `git restore`/`reset --hard` of journaled paths
  destroys uncommitted local edits — worst in Phase 2, where BerryWiki never
  commits, so "HEAD" predates all user work. Redesign recovery to never touch
  unrelated dirty state (e.g. operate only on temp files + atomic rename,
  reconcile forward, never hard-reset the working tree).
. *App-private state must not live inside the clone.* `.berrywiki/journal.json`
  (and any lock/state) inside the canonical wiki clone can be committed and
  pushed by a plain-git user — a direct "usable without BerryWiki" breach, and
  it contradicts the plan's own never-inside-the-clone rule for the index and
  drafts. Move journal/lock/state to the XDG cache location keyed by repo
  identity, as already done for the search index.
. *Fine-grained PAT vs `.wiki.git` is unverified.* The credential design bakes
  in fine-grained-PAT access to the wiki repo, historically unsupported by
  GitHub, and only exercises it at S3.5 — after the whole sync engine is built
  on it. Add an auth row to the *read-side* spike (Phase 1) that actually
  clones a private `.wiki.git` with the chosen credential, and treat the token
  type as unverified until then (ADR-0002 already gates this).
. *No stale-write / external-modification guard.* Phase 2's operating model is
  a running server plus concurrent terminal git in the same working tree, yet
  nothing detects that a page changed on disk between read and write. Add
  mtime/hash precondition checks (compare-and-swap on save) and a reload path.
. *Guix packaging is under-sized and CI-unverified.* The estate-primary Guix
  path is one Phase-5 slice but is MVP-gating; `git` as a "propagated input"
  is the wrong mechanism for a binary's runtime dependency, and packaging the
  full axum+tokio crate tree under `cargo-build-system` is a real cost with no
  risk-register row — while CI only ever builds the Nix fallback. Give Guix
  its own slice + risk row, and add a Guix build lane to CI.
. *Silently dropped spec items.* Backup/export (upstream Phase 4), the
  controlled rich editor, and the command palette appear in no phase and no
  non-goals list. Record each explicitly as either a scheduled slice, a
  post-MVP backlog item, or a declared non-goal.

=== Minor — should address

* *Zero-JS gate under-enforces.* Asserting "no `<script>`" misses inline
  `on*=` handlers and `javascript:` URLs, and collides with GitHub-accepted
  inline wiki HTML. State comrak's raw-HTML policy (`unsafe_` off, or
  sanitised) and enforce a real HTML-safety check, not a substring scan that
  false-positives on escaped code-fence examples.
* *Quiet MVP redefinitions.* "Link autocomplete" (a `<datalist>` side picker,
  not inline) and "remote-change detection" (a manual fetch button, nothing
  proactive) are narrowed from the MVP checklist, yet Phase 5 declares "every
  checklist item verified". Record the re-scope, or deliver the checklist item.
* *Leaf-only delete* is narrowed from the spec's unqualified delete without a
  recorded scope decision (no subtree-delete / reparent-then-delete flow).
* *No error-quality or observability requirements* anywhere, despite the spec's
  actionable-errors and structured-logging sections. Add them.
* *Demote-or-gate language slip:* "provably/proven" is used for
  merely harness-tested behaviour; reserve it for a gating `gnatprove` lane.
* *AsciiDoc-by-default slip:* the S5.2 import report is specified as Markdown;
  technical reports are AsciiDoc (Markdown is for wiki content only; AsciiDoc for community-health files).
* *Grep gates false-positive:* the `--force`/`reset --hard` source scan will
  match the force-push audit test and the command-enum docs (tests live in
  `src/` here). Scope the gate to exclude tests/docs or assert on the enum.
* *Textarea CRLF:* browsers normalise textarea newlines to CRLF; without a
  stated newline policy the "zero-byte diff" round-trip criterion will churn.
* *GIT_ASKPASS* answers both username and password prompts with the token; fix
  the username path rather than forbidding it.
* *Stale repo facts:* the executive summary cites "four commits" and "six
  conflict tests" — the repo has more (and `conflicts.rs` has 7 `#[test]`s).
  Refresh before ratifying.
