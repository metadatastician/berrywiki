== Filename strategy findings

See link:../decisions/0001-page-filename-strategy.adoc[ADR-0001]. Working
hypothesis given "native reader must work": flat `--`-separated filenames
(`Teaching--Course-A--Assessment-Plan.md`), because the native wiki UI does not
present real directory hierarchy. *Unverified* until the nested-directory test
runs.

== Rendering fidelity (BerryWiki preview vs GitHub)

BerryWiki renders its preview with comrak GFM (`berrywiki-render`). Known,
deliberate differences from GitHub's wiki pipeline тАФ all *safety-over-fidelity*
and none affecting the canonical source:

[cols="3,2", options="header"]
|===
| Aspect | BerryWiki policy
| Raw inline/block HTML | *Escaped/omitted*, never emitted (comrak `unsafe_=false` + `tagfilter`). GitHub renders a *sanitised subset* of inline HTML, so some GitHub-rendered HTML will not appear in BerryWiki's preview. тЬЕ tested.
| `javascript:` / `vbscript:` / non-image `data:` link URLs | Rewritten to `#` at the AST level. тЬЕ tested.
| Syntax highlighting of code blocks | Not applied (comrak default features off to stay buildable on rustc 1.85). May return behind a feature.
| Exact GFM parity (tables, task lists, autolinks, footnotes) | comrak GFM; *parity with GitHub's exact output is UNVERIFIED* until the live spike (тЭУ).
|===

Source remains canonical regardless; the preview is advisory.

== Open questions

* Exact anchor slug algorithm GitHub uses for headings (we implement a
  GitHub-compatible approximation; confirm edge cases with Unicode).
* Whether `_Sidebar.md` supports nested lists in the rendered sidebar to the
  depth we generate.
* Attachment path resolution relative to page location after sync.

== Recommendations (provisional, pending the spike)

. Persist links in GitHub-native Markdown; accept `[[тАж]]` only as *authoring*
  input, serialised on save.
. Use flat `--` filenames; keep the tree in metadata, never infer it from paths.
. Keep the metadata HTML comment, but verify invisibility on a live page before
  relying on it.
